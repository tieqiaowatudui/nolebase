# 理解系统
Code助手加快理解系统.当一个新手接收一个项目的时候,可以使用Code助手,帮助定位模块,梳理核心流程,排查bug出现位置等.
常用的语法包括:
> 用户身份论证的逻辑在当前仓库中的哪个位置?
> 用户下单请求到最终交易完成的核心逻辑包括哪些?
> 当前仓库中,哪些模块和订单模块交易,出现下单错误的场景有哪些?

# 重构和迁移
通过Code助手,能够很方便的整理出重构的功能,并且添加上必要的测试用例,同时也非常方便的列出系统迁移等工作的关键点.
常用的语法包括:
> 将当前文件拆分成多个模块,同时为每个模块添加上必要的测试用例
> 将访问数据库中同步逻辑,改成asyn/awit的实现方式.

# 优化系统性能
让Code助手,查看当前软件下单流程中的存在系统瓶颈的地方,以及解决这些瓶颈的方式.
常用语法:
> 帮忙分析一下整个下单流程中的内存消耗的地方?应该如何优化
> 当前函数的功能实现步骤是什么?线上运转内存消耗较高.请帮忙分析一下如何优化,是否需要添加缓存来实现?
> 当期那逻辑中哪些步骤是不需要的,哪些步骤可以合并,以获取到更加优秀的查询效率?

# 提升测试覆盖率
测试用例的编写一直都是大家的难点.用Code助手,帮忙写测试用例,并且运转测试用例.并生成测试报告.
常用语法:
> 帮忙当前包中的所有函数都生成测试用例,尤其是边界条件.失败的逻辑都要覆盖到.测试完成生成一个测试报告.

# 加快开发进度
使用Code助手,一方面在开始的时候帮助生成工程结构,基础的类和函数,同时实现最基本的接口功能,例如参数校验,日志等.另一方面,填充最底层的代码.就是最后一公里的代码让Code助手帮助实现.
常用语法:
> 帮忙实现ugc的写入,按照某几个参数进行查询的接口,同时生成参数的非空校验以及日志.
> 这是一个项目工程的tree结构.请帮忙生成这样结构的目录.以shell脚本的形式提供.

# 保持专注
由于日常中,有很多工作任务,这个时候就需要上下文进行切换了.使用Code助手,只要将一件事创建一个Code助手的上下文.完成后就将这个任务标记为完成就好
常用语法:
> 帮我分析一下当前下单流程的核心步骤,最终结果记录下来,我后续来查看.
> 当前的内容先总结一下,我后续处理.

# 创意与探索
让Code助手帮忙做一些开放的工作.例如设计方案,验证结果,检查类似问题等等
常用语法:
> 帮忙查看一下整个工程中,使用手动sql的地方,而不是使用查询器的地方.
> 系统使用事件驱动开发,而不是请求响应的架构会如何运转?
> 当前架构中,订单模块帮忙拆分成更小模块,并且陈诉思考过程.
